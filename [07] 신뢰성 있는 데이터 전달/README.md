## 카프카 핵심 가이드

### 신뢰성 있는 데이터 전달

신뢰성은 카프카 단일 컴포넌트의 속성이 아니라 전체 시스템의 속성

브로커 설정, 토픽 설정, 프로듀서 설정, 컨슈머 처리 방식, 장애 검증이 함께 맞아야 함

카프카는 처리량, 지연시간, 가용성, 저장 비용 사이에서 신뢰성 수준을 조절할 수 있도록 설계

<br>

### 카프카가 제공하는 신뢰성 보장

**기본 보장**
- 같은 프로듀서가 같은 파티션에 A 이후 B를 쓰면 B의 오프셋이 더 큼
- 컨슈머는 같은 파티션 안에서 오프셋 순서대로 메시지를 읽음
- 메시지는 모든 ISR에 기록된 뒤 커밋된 것으로 간주
- 커밋된 메시지는 최소 1개의 레플리카가 살아있는 한 유실되지 않음
- 컨슈머는 커밋된 메시지만 읽을 수 있음

<br>

**주의사항**
- 카프카의 기본 보장만으로 전체 애플리케이션의 신뢰성이 자동으로 보장되는 것은 아님
- 프로듀서가 성공 응답을 언제 받을지, 컨슈머가 오프셋을 언제 커밋할지에 따라 유실/중복 가능성이 달라짐
- 신뢰성을 높이면 일반적으로 지연시간, 처리량, 저장 비용, 운영 복잡도 중 일부를 감수해야 함

<br>

### 복제와 ISR

카프카 신뢰성의 핵심은 파티션 복제

각 파티션은 1개의 리더 레플리카와 0개 이상의 팔로워 레플리카를 가짐

프로듀서 쓰기는 리더에게 전달되고, 팔로워는 리더의 로그를 따라가며 복제

리더가 장애가 나면 ISR 안의 레플리카 중 하나가 새 리더가 됨

<br>

**ISR(In-Sync Replicas)**
- 리더와 동기화 상태를 유지하는 레플리카 집합
- 리더와 연결되어 있고, 정해진 시간 안에 데이터를 따라잡고 있어야 함
- 뒤처진 레플리카는 ISR에서 제외
- 다시 최신 메시지까지 따라잡으면 ISR에 재포함

<br>

**ISR이 줄어들 때의 의미**
- 복제 팩터가 3이어도 ISR이 1개뿐이면 실제 안전 여유는 1개뿐
- ISR이 적을수록 장애 시 데이터 유실 또는 파티션 비가용 위험이 커짐
- ISR이 많은 상태를 유지하려면 네트워크, 디스크, GC, 브로커 부하를 함께 관리해야 함

<br>

### 브로커와 토픽 설정

신뢰성에 직접적인 영향을 주는 대표 설정:

- `replication.factor`
- `unclean.leader.election.enable`
- `min.insync.replicas`

<br>

### 복제 팩터

`replication.factor`: 토픽 파티션을 몇 개의 레플리카로 저장할지 결정

`default.replication.factor`: 자동 생성 토픽에 적용되는 기본 복제 팩터

<br>

**복제 팩터 N의 의미**
- 최대 N-1개의 브로커 장애까지 읽기/쓰기 가능성을 확보
- 데이터가 N개 복사본으로 저장되므로 디스크 사용량 증가
- 팔로워 복제 트래픽이 늘어나므로 브로커 간 네트워크 사용량 증가

<br>

**복제 팩터 결정 기준**
- 가용성: 브로커 재시작이나 장애 중에도 파티션을 사용할 수 있어야 함
- 내구성: 디스크나 브로커 장애에도 데이터 복사본이 남아야 함
- 처리량: 복제본이 많을수록 브로커 간 복제 트래픽 증가
- 비용: 복제본 수만큼 저장소와 네트워크 비용 증가
- 장애 도메인: `broker.rack`을 설정하여 랙/가용영역 단위 장애에 대비

<br>

### 언클린 리더 선출

`unclean.leader.election.enable`: ISR 밖의 레플리카를 리더로 선출할 수 있는지 결정

<br>

**false**
- ISR에 남은 레플리카가 없으면 파티션을 오프라인 상태로 둠
- 데이터 유실 가능성을 줄이는 방향
- 가용성은 낮아질 수 있음

<br>

**true**
- ISR 밖의 뒤처진 레플리카도 리더가 될 수 있음
- 파티션을 더 빨리 사용 가능하게 만들 수 있음
- 기존 리더에만 있던 메시지는 유실될 수 있음
- 같은 오프셋에 서로 다른 메시지가 존재했던 것처럼 보이는 불일치가 발생할 수 있음

<br>

운영 기본값은 데이터 유실을 피하기 위해 `false`를 기준으로 잡는 것이 안전

<br>

### 최소 ISR

`min.insync.replicas`: 쓰기 성공으로 인정받기 위해 필요한 최소 ISR 수

프로듀서가 `acks=all`로 전송할 때 의미가 커짐

<br>

예시:
```text
replication.factor=3
min.insync.replicas=2
acks=all
```

위 설정에서는 3개 레플리카 중 최소 2개가 ISR에 있어야 쓰기가 성공

ISR이 1개만 남으면 브로커는 쓰기 요청을 거절하고, 컨슈머는 이미 커밋된 데이터만 계속 읽을 수 있음

<br>

**장점**
- 1개의 레플리카에만 기록된 메시지가 성공 처리되는 상황을 방지
- 장애 상황에서 데이터 유실 가능성을 줄임

**단점**
- ISR이 부족하면 쓰기 가용성이 떨어짐
- `NotEnoughReplicasException` 또는 관련 예외를 처리해야 함

<br>

### 레플리카 동기화 유지

Out-of-sync replica가 많아질수록 실제 복제 여유가 줄어듦

`replica.lag.time.max.ms`: 팔로워가 리더의 최신 메시지를 정해진 시간 안에 따라잡지 못하면 ISR에서 제외

ZooKeeper 기반 클러스터에서는 `zookeeper.session.timeout.ms`가 브로커 생존 판단에 영향

KRaft 기반 클러스터에서는 컨트롤러 쿼럼 설정이 관련되므로 운영 Kafka 버전에 맞춰 확인 필요

<br>

### 디스크 저장과 fsync

카프카는 메시지를 매번 디스크에 강제 fsync하지 않고, OS 페이지 캐시와 복제를 적극 활용

브로커 장애보다 서로 다른 브로커/랙의 복제본을 유지하는 것이 더 효율적인 내구성 전략

<br>

관련 설정:
- `flush.messages`: 디스크 동기화 전 허용할 메시지 수
- `flush.ms`: 디스크 동기화 주기

<br>

강제 fsync를 자주 수행하면 내구성 체감은 올라갈 수 있지만 처리량은 크게 떨어질 수 있음

일반적으로는 복제 팩터, ISR, `acks` 조합을 먼저 설계

<br>

### 프로듀서 신뢰성

브로커를 안전하게 설정해도 프로듀서 설정이 약하면 메시지는 유실될 수 있음

프로듀서에서 중요한 두 가지:
- 요구사항에 맞는 `acks` 설정
- 재시도 가능한 오류와 재시도 불가능한 오류 처리

<br>

### acks 설정

`acks=0`
- 프로듀서가 브로커 응답을 기다리지 않음
- 지연시간은 낮지만 메시지가 실제로 저장되었는지 알 수 없음
- 신뢰성이 중요한 데이터에는 부적합

<br>

`acks=1`
- 리더 레플리카에 기록되면 성공 응답
- 팔로워 복제가 끝나기 전에 리더가 장애 나면 메시지 유실 가능
- 처리량과 지연시간을 신뢰성보다 더 중시할 때 사용 가능

<br>

`acks=all`
- 모든 ISR에 기록된 뒤 성공 응답
- `min.insync.replicas`와 함께 사용할 때 내구성 보장 강화
- 가장 안전하지만 지연시간은 증가

<br>

### 프로듀서 재시도와 오류 처리

`retries`: 재시도 가능한 오류 발생 시 재전송

`delivery.timeout.ms`: 메시지 전송을 포기하기 전까지 허용할 전체 시간

`enable.idempotence`: 재시도로 인한 중복 쓰기를 방지

<br>

**재시도 가능한 오류**
- 리더 선출 중
- 일시적 네트워크 문제
- 브로커가 잠시 응답하지 않음

**재시도 불가능한 오류**
- 직렬화 실패
- 권한 오류
- 메시지 크기 초과
- 잘못된 설정

<br>

프로듀서에서 오류를 단순 로그로만 남기면 애플리케이션 관점에서는 메시지가 유실될 수 있음

요구사항에 따라 중단, 재시도, 로컬 저장, 별도 실패 큐 전송, 백프레셔 적용 중 하나를 선택해야 함

<br>

### 컨슈머 신뢰성

컨슈머는 커밋된 메시지만 읽기 때문에 읽는 데이터 자체는 일관됨

중요한 문제는 "어디까지 처리했는지"를 정확히 기록하는 것

컨슈머가 처리하지 않은 메시지의 오프셋을 커밋하면 메시지 유실

처리한 메시지의 오프셋을 커밋하지 못하면 메시지 중복 처리

<br>

### 컨슈머 주요 설정

`group.id`
- 같은 그룹의 컨슈머들은 파티션을 나눠 읽음
- 각 애플리케이션이 모든 메시지를 독립적으로 읽어야 하면 고유한 그룹 ID 사용

<br>

`auto.offset.reset`
- 커밋된 오프셋이 없거나 유효하지 않을 때 시작 위치 결정
- `earliest`: 처음부터 읽어 유실 가능성을 낮추지만 중복 처리 가능성 증가
- `latest`: 최신 위치부터 읽어 중복은 줄지만 과거 메시지를 놓칠 수 있음
- `none`: 유효한 오프셋이 없으면 예외 발생

<br>

`enable.auto.commit`
- `true`: 컨슈머가 주기적으로 오프셋 자동 커밋
- `false`: 애플리케이션 코드에서 명시적으로 커밋

<br>

`auto.commit.interval.ms`
- 자동 커밋 주기
- 주기가 짧으면 중복 가능성은 줄지만 커밋 오버헤드 증가
- 주기가 길면 오버헤드는 줄지만 장애 시 중복 처리 범위 증가

<br>

### 수동 오프셋 커밋

신뢰성이 중요한 처리는 수동 커밋을 고려

핵심 원칙:
- 메시지 처리가 끝난 뒤 오프셋 커밋
- 읽은 오프셋이 아니라 처리 완료한 다음 오프셋을 커밋
- 리밸런스 전에 현재 처리 상태와 오프셋 정리

<br>

**커밋 빈도**
- 자주 커밋하면 장애 시 중복 처리는 줄어듦
- 커밋은 내부 토픽에 쓰는 작업이므로 너무 자주 하면 성능 저하
- 메시지마다 커밋하는 방식은 낮은 처리량 토픽이 아니면 비효율적

<br>

### 컨슈머 재시도 패턴

카프카 컨슈머는 개별 메시지를 ack하지 않고 오프셋을 커밋

따라서 중간 메시지 처리에 실패한 상태에서 이후 오프셋을 커밋하면 실패 메시지가 건너뛰어질 수 있음

<br>

**패턴 1: pause 후 재처리**
- 성공적으로 처리한 마지막 오프셋까지만 커밋
- 실패한 레코드는 메모리나 별도 저장소에 보관
- `pause()`로 추가 poll 데이터를 막고 실패 레코드 재처리
- 성공하면 `resume()` 후 다시 소비

<br>

**패턴 2: 재시도 토픽**
- 처리 실패 레코드를 별도 retry topic에 기록
- 메인 토픽 소비는 계속 진행
- 별도 컨슈머 그룹이 retry topic을 처리
- 최종 실패는 dead letter topic으로 분리 가능

<br>

### 상태를 가진 컨슈머

윈도우 집계, 이동 평균, 조인처럼 poll 사이에 상태를 유지하는 처리는 더 복잡

오프셋만 커밋하고 상태를 잃으면 재시작 후 결과가 틀어질 수 있음

상태와 오프셋을 함께 복구할 수 있어야 함

<br>

해결 방향:
- 처리 결과와 오프셋을 함께 저장
- 상태 저장소를 별도로 두고 복구 절차 정의
- 복잡한 상태 처리는 Kafka Streams, Flink 같은 상위 수준 라이브러리 고려

<br>

### 신뢰성 검증

신뢰성은 설정했다고 끝나는 것이 아니라 검증해야 함

검증 계층:
- 설정 검증
- 애플리케이션 검증
- 운영 모니터링

<br>

### 설정 검증

카프카는 검증용 도구를 제공

- `VerifiableProducer`
- `VerifiableConsumer`

<br>

검증 시나리오:
- 리더 브로커 종료 후 복구 확인
- 컨트롤러 재시작 후 정상 동작 확인
- 브로커 롤링 재시작 중 메시지 유실 여부 확인
- ISR 부족 상황에서 쓰기 거절 동작 확인
- 언클린 리더 선출 허용 시 데이터 유실 가능성 확인

<br>

생산된 메시지 수와 소비된 메시지 수를 비교하여 유실/중복 여부를 확인

<br>

### 애플리케이션 검증

애플리케이션은 Kafka 클라이언트 설정 외에도 자체 오류 처리 로직을 검증해야 함

검증 대상:
- 프로듀서 예외 처리
- 컨슈머 오프셋 커밋 시점
- 리밸런스 리스너
- 재시도/실패 큐 처리
- 상태 복구 로직

<br>

장애 조건:
- 브로커 연결 끊김
- 브로커와 클라이언트 사이의 높은 지연시간
- 디스크 full
- 리더 선출
- 브로커 롤링 재시작
- 컨슈머/프로듀서 롤링 재시작

<br>

### 운영 모니터링

프로덕션에서는 클러스터뿐 아니라 클라이언트와 데이터 흐름까지 모니터링 필요

<br>

**프로듀서 지표**
- error-rate
- retry-rate
- 전송 실패 로그
- timeout 발생 빈도

<br>

**컨슈머 지표**
- consumer lag
- poll 지연
- 처리량
- 리밸런스 빈도

<br>

**브로커 지표**
- `FailedProduceRequestsPerSec`
- `FailedFetchRequestsPerSec`
- under-replicated partitions
- offline partitions

<br>

단순히 lag가 순간적으로 증가하는 것보다, 시간이 지나도 계속 따라잡지 못하는지가 더 중요

생산량과 소비량을 함께 비교해야 전체 데이터 흐름의 신뢰성을 판단할 수 있음

<br>

### 운영 관점 체크 포인트

- 중요한 데이터: `replication.factor=3`, `min.insync.replicas=2`, `acks=all` 조합 고려
- 데이터 유실 방지 우선: `unclean.leader.election.enable=false`
- 중복 허용 불가: 멱등성 프로듀서와 트랜잭션 검토(8장)
- 컨슈머 유실 방지: 처리 완료 후 오프셋 커밋
- 장애 대응: 설정값을 문서화하고 장애 시나리오를 반복 검증
